# マナタネ (manatane)
## プロトタイプ実装用仕様書 v0.3

---

## 1. 実装目的

サービス名はカタカナで「マナタネ」、英字表記は半角英数小文字で「manatane」とする。これら以外の表記(MANATANE、Manatane、ManaTane等)は使用しない。

「マナタネ」は、「学びの種」を意味する造語である。

スマートフォン向けの学習入口サービス「マナタネ」の動作プロトタイプを作成する。本サービスは、ユーザーのその日の目的・気分・興味に合わせて、学び始めるための入口を提案するサービスである。

本v0.3における主要な変更点は四つである。第一に、診断結果として提示するresult_idを5種類から81種類に拡張する。第二に、ガイドキャラクターを5名から81名に拡張する(ただしresult_idとは独立した次元で判定する)。第三に、ミニゲームを「うっかりチェック」から知的ゲーム2種類(スライドパズル4×4、迷路8×8)に置き換える。第四に、診断結果は新4軸(関心の対象・知ろうとする対象・変えたい方向・関わり方の深さ)の組み合わせで定義する。

プロトタイプの位置づけは「ある程度プロトタイプのままで本番になる可能性があるもの」とする。本仕様書では会員登録、ログイン、決済、DB保存、外部API連携、本格アフィリエイト計測は実装しないが、ガイド選定にあたっては将来のアフィリエイト導線(関連書籍の流通)を前提に選定する。

このプロトタイプの目的は、稟議・企画検討時に以下を確認できる状態にすることである。ユーザーが1分程度で診断を体験できる、診断結果として「今日の学び方の入口」が表示される、ガイドキャラクターが表示される、ガイド詳細ページから別の学習機会へつながる、おすすめ学習ルートが表示される、スマートフォン専用UIとして自然に操作できる、の六点である。

## 2. 実装範囲

### 2.1 実装する機能

スマートフォン専用UI、ホーム画面、今日の目的選択画面、診断形式選択画面、ゲーム選択画面、クイズ診断、今の気分から見つける（診断形式mood）、気になることから探す（診断形式topic）、これまでの学びから探す（診断形式path）、ミニゲーム診断(スライドパズル4×4、迷路8×8)、診断結果画面、今日のガイド表示、ガイド詳細画面、おすすめルート画面、保存案内画面、localStorageによる一時保存、JSONデータによる診断項目・結果・ガイド・ルート管理を実装する。

新4軸による81種類のresult_id判定ロジック、81名のガイドとの独立次元判定(ハイブリッド方式)、ゲーム行動から行動傾向タグへの変換も含む。

### 2.2 実装しない機能

会員登録、ログイン、メール認証、決済、DB保存、外部API連携、本格アフィリエイト計測、管理画面、AI API連携、プッシュ通知、本格的なアクセス解析は実装しない。

## 3. 表記ルール

サービス名の日本語表記は「マナタネ」とする。英字表記は `manatane` とする。HTMLのtitleタグ、metaタグ、README、localStorageキー、CSS/JSコメント内でも、英字表記は `manatane` に統一する。MANATANE、Manatane、ManaTane などの表記は使用しない。READMEのタイトルは `# マナタネ (manatane) プロトタイプ` とする。localStorageの保存キーは `manatane_state` とする。

## 6. デザイン方針

### 6.1 基本方針

スマートフォン専用、1画面1アクション、カード型UI、タップ中心とする。横スクロールは原則使わない、長文を避ける、結果はカードで表示する、下部固定ナビを配置する、PCで開いた場合もスマホ幅で中央表示する。

### 6.2 画面幅

```css
body {
  max-width: 480px;
  margin: 0 auto;
}
```

### 6.3 ボタン

高さは最低48px、推奨高さは56px〜64px、横幅いっぱいのカード型ボタン、タップしやすい余白を確保する。

### 6.4 色・印象

白または淡い背景、淡いブルー・グリーン・ベージュ系のアクセント、角丸カードを基本とする。過度なアニメーションは禁止。占いサイト、高額スクールLP、副業煽りLPのような印象を避ける。

### 6.5 表示名の長さに関する注意

新仕様の表示名(例「考えて自分に触れ、見方を見直す入口」)は15〜22文字と長い。結果カードのタイトル領域では2行表示を許容する。フォントサイズは見出しレベルで18〜20pxを推奨する。長すぎる場合は省略せず、改行して全文表示する。

## 9. 画面仕様

---

### 9.1 ホーム画面

#### 目的

ユーザーに軽く開始してもらう。

#### 表示内容

サービス名は「マナタネ」、メインコピーは「今日、何を少し学んでみる?」、説明文は「いくつかの質問に答えると、今日の学びの入口が見つかります。」、ボタンは「1分で始める」、補足は「診断結果は固定的なタイプ判定ではなく、今日の学び方の参考です。」とする。

#### ボタン動作

「1分で始める」ボタンで今日の目的選択画面へ遷移する。

---

### 9.2 今日の目的選択画面

#### 目的

ユーザーの今日の状態を取得する。これにより、後段でのresult_id判定のうち軸A・軸Bの推定に利用する。

#### 表示内容

タイトルは「今日学びたいのは、どんなことですか?」、選択肢は6つである。生活に役立てたい、知的好奇心を満たしたい、趣味を深めたい、副業の種を探したい、苦手を減らしたい、今日は眺めるだけ、を提示する。

#### purpose_id

各選択肢に対応するpurpose_idは以下の通りである。「生活に役立てたい」はlife、「知的好奇心を満たしたい」はcuriosity、「趣味を深めたい」はhobby、「副業の種を探したい」はsidejob、「苦手を減らしたい」はweakness、「今日は眺めるだけ」はbrowseとする。

#### ボタン動作

選択後、選んだpurpose_idを状態に保存し、診断形式選択画面へ遷移する。

---

### 9.3 診断形式選択画面

#### 目的

ユーザーが入口形式を選べるようにする。

#### 表示内容

タイトルは「どれで試しますか?」、タイトル直下に橋渡しの説明文「どれを選んでも結果は今日のあなたに合わせて出ます。」を表示し、その下に5つの選択肢を並べる。選択肢はクイズで試す(1〜3問で興味や理解を軽く確認します)、今の気分から見つける(今日の気分についての短い質問から、学び方を見つけます)、気になることから探す(最近の興味から学び方を見つけます)、これまでの学びから探す(これまで学んできたことから、学び方を見つけます)、ゲームで試す(知的ゲームから入口を探します)である。なお内部の diagnosis_type 値は、「今の気分から見つける」は mood、「気になることから探す」は topic、「これまでの学びから探す」は path として保存する。topic と path は4問×4軸を直感的な選択でたどる経路(セクション9.11・9.12)で、quiz・mood とは独立の判定ロジックを用いる。

#### ボタン動作

「クイズで試す」は diagnosis_type を quiz として保存し、クイズ診断画面へ遷移する。「今の気分から見つける」は diagnosis_type を mood として保存し、今の気分から見つける画面へ遷移する。「気になることから探す」は diagnosis_type を topic として保存し、気になることから探す画面(セクション9.11)へ遷移する。「これまでの学びから探す」は diagnosis_type を path として保存し、これまでの学びから探す画面(セクション9.12)へ遷移する。「ゲームで試す」は diagnosis_type を game として保存し、ゲーム選択画面へ遷移する。

---

### 9.4 クイズ診断画面

#### 目的

短いクイズで興味、自信度、理解状態を取得する。回答時間と自信度から、軽い行動傾向タグ推定も行う。

#### 表示内容

タイトルは「1分クイズ」、現在の問題番号、問題文、選択肢4つ、回答後に自信度選択を表示する。

#### クイズ数

プロトタイプでは3問出題する。

#### 出題方法

quizzes.json から、今日の目的に近いジャンルの問題を3問取得する。該当ジャンルがない場合はランダムに3問取得する。

#### 回答後の流れ

回答を選ぶ、正誤を表示する、自信度を選ばせる、次の問題へ進む、3問終了後に結果画面へ遷移する。

#### 自信度選択肢

「かなりある」「たぶんある」「あまりない」「勘」の4つから選ぶ。

#### 保存する情報

quiz_id、selected_answer_id、is_correct、confidence、response_seconds(回答秒数)を保存する。

#### スコア算出ルール

ユーザーが選んだ選択肢の scores を is_correct の値に関わらず加算する。理由は、quizzes.json では不正解の選択肢にも scores が設定されており、これは不正解でもその選択肢が示すユーザーの傾向を反映する意図のためである。confidence は記録のみ行い、result_id 判定スコアには寄与しない。本番版では confidence を判定に組み込む可能性があるが、判定式はプロトタイプ確認後に別途設計する。

#### 行動傾向タグの軽推定

クイズ診断では、選択した回答の confidence と response_seconds から行動傾向タグを軽く推定する。confidence が「かなりある」かつ response_seconds が3秒未満の場合は quick_action タグを付与する。confidence が「勘」の場合は intuition タグを付与する。response_seconds が10秒以上の場合は cautious タグを付与する。これらのタグはガイド判定の補助情報として使用する。

---

### 9.5 今の気分から見つける画面

#### 目的

軽い質問で今日の学び方を推定する。3問の選択肢それぞれに、result_id 判定用の scores と行動傾向タグの両方を紐づける。

#### 表示内容

タイトルは「今日の学び方チェック」、質問数は3問、各質問は質問文と選択肢3〜5個で構成する。

#### 質問例と行動傾向タグの紐づけ

Q1「新しいことを学ぶなら、今日はどれが近い?」の選択肢は4つである。「まず試したい」は quick_action タグ、「全体像を知りたい」は planning タグ、「手順があると安心」は cautious タグ、「気になるものを眺めたい」は exploration タグを紐づける。

Q2「今の学習に使える時間は?」の選択肢は4つである。「1分だけ」は軸D=D1に、「5分くらい」は軸D=D1〜D2に、「15分くらい」は軸D=D2に、「今日は見るだけ」は軸D=D1かつ persistence タグの逆方向(switching)に紐づける。

Q3「つまずいた時に近い行動は?」の選択肢は4つである。「調べる」は exploration タグ、「誰かに聞く」は dialogue タグ、「一度止まる」は cautious タグ、「別の方法を探す」は switching タグを紐づける。

#### 回答後

各回答のresult_id用スコアを合算し、行動傾向タグも集計して、結果画面へ遷移する。

---

### 9.6 ゲーム選択画面・ミニゲーム診断画面

#### 9.6.1 ゲーム選択画面

##### 目的

ユーザーがプレイするゲームを2種類から選ぶ。

##### 表示内容

タイトルは「どちらのゲームで試しますか?」、選択肢は2つである。スライドパズル(4×4のピースを順に並べるパズル、約30〜60秒)、迷路(8×8の経路をゴールまで進む、約30〜60秒)を提示する。

##### ボタン動作

「スライドパズル」を選んだ場合は game_type を slide_puzzle として保存し、スライドパズル診断画面へ遷移する。「迷路」を選んだ場合は game_type を maze として保存し、迷路診断画面へ遷移する。

---

#### 9.6.2 スライドパズル4×4診断画面

##### ルール

4×4の盤面に15個のピース(番号1〜15)と1個の空きマスが配置される。隣接するピースを空きマスにスライドさせ、最終的に1から15まで順に並べる。制限時間は120秒とする。タイムアウトまたはユーザーの中断で終了する。

##### 表示

タイトルは「スライドパズル」、説明は「ピースをタップして1から15まで順に並べてください。」、ゲームエリアに4×4の盤面、残り時間カウントダウン、現在の手数を表示する。

##### 初期配置

完成形からランダムに20手戻した状態を初期配置とする。理論的に解ける配置のみを生成する。

##### 取得データ

取得する指標は以下の通りである。moves(完成までの手数)、optimal_moves(理論最小手数、初期配置から計算)、efficiency_ratio(moves ÷ optimal_moves)、backtrack_count(同一ピースを2回以上動かした回数)、elapsed_seconds(所要秒数)、completed(完成したか)、give_up_count(リセット回数)、first_move_seconds(最初の操作までの秒数)である。

##### 結果変換

取得したデータから行動傾向タグへの変換は、セクション10.7に従う。result_id判定への影響もセクション10.7で定義する。

##### 補足表示

ゲーム終了画面には以下の文言を表示する。「このゲーム結果は、今日の学び方を考えるための参考情報です。能力を確定するものではありません。」

---

#### 9.6.3 迷路8×8診断画面

##### ルール

8×8の格子状の迷路で、スタート地点からゴール地点までを進む。壁に阻まれる箇所では別ルートを探す。制限時間は120秒とする。タイムアウトまたはユーザーの中断で終了する。

##### 表示

タイトルは「迷路」、説明は「スタートからゴールまで進んでください。」、ゲームエリアに8×8の迷路、残り時間カウントダウン、現在の所要時間を表示する。

##### 操作方法

スマートフォン操作のため、画面下部に上下左右の方向ボタンを配置する。または、迷路セル上でのスワイプ操作も受け付ける。

##### 迷路生成

毎回ランダムに迷路を生成する。ただし、スタートからゴールまでの解が必ず存在することを保証する。

##### 取得データ

取得する指標は以下の通りである。elapsed_seconds(ゴール到達秒数)、path_length(通過した経路の長さ)、optimal_path_length(最短経路の長さ)、detour_ratio(path_length ÷ optimal_path_length)、dead_end_count(行き止まりに進入した回数)、backtrack_cells(戻り移動したセル数)、completed(ゴール到達したか)、give_up_count(諦め回数)、first_move_seconds(最初の操作までの秒数)である。

##### 結果変換

取得したデータから行動傾向タグへの変換は、セクション10.7に従う。

##### 補足表示

ゲーム終了画面には以下の文言を表示する。「このゲーム結果は、今日の学び方を考えるための参考情報です。能力を確定するものではありません。」

---

### 9.7 診断結果画面

#### 目的

今回の診断結果を表示する。

#### 表示方針

ユーザーを固定的に分類しない。必ず「今日の入口」「今日の学び」「今日の探究」のいずれか(軸Dに対応)という表現を使う。

#### 表示内容

タイトルは「今日の学び方の入口」、結果名は81種類のうち1件の表示名(例「考えて自分に触れ、見方を見直す入口」)、説明はresults.jsonの description フィールド、おすすめは recommendations フィールドから3項目、避けたい入口は avoid フィールドから2項目を表示する。

#### 表示ボタン

「今日のガイドを見る」「おすすめルートを見る」「もう一度診断する」の3つを表示する。

#### ボタン動作

「今日のガイドを見る」はガイド詳細画面へ遷移、「おすすめルートを見る」はおすすめルート画面へ遷移、「もう一度診断する」は今日の目的選択画面へ遷移する。

---

### 9.8 ガイド詳細画面

#### 目的

診断結果に対応するガイド(ハイブリッド方式で決定)を表示し、別の学習機会を作る。

#### 表示内容

タイトルは「今日、気が合いそうなガイド」、ガイド名は guides.json の name、参考人物は name_original、フレーム名は frame_name、表示ラベルは display_label、一言は short_line、注意書きは「この一言は本人の発言ではなく、考え方を学習向けに要約したものです」を固定表示、略歴は bio_short、この人から借りる視点は learning_translation、今日の1分アクションは one_min_action、関連テーマは related_topics(3〜4個)、生没年は era、地域は region を表示する。

### 9.11 気になることから探す画面（diagnosis_type=topic）

#### 目的

ユーザーが「最近の興味・関心」を切り口に、4つの問いに直感で答えることで4軸すべてを決定し、result_id 81通りのいずれにも到達しうる入口を提供する。クイズや気分質問とは異なり、正解や攻略を考えさせず、感覚で選べる短い設問の連続によって、軽い認知負荷のまま多様な結果に分岐させる。

#### 表示内容

タイトルは「気になることから探す」、その下に進捗（「質問 N / 4」）、現在の問い、選択肢を縦に並べる。問いと選択肢は4問のシーケンスで構成される。各問が4軸のうち1つに対応し、選択肢の数は問いごとに異なってよいが、対応する軸の3つの値(例: A1/A2/A3)すべてが少なくともひとつの選択肢でカバーされている必要がある。本仕様のプロトタイプ実装では各問とも3選択肢で軸の3値に1対1対応とする。

問いと選択肢、対応する軸値の一覧は以下の通り。

| 問 | 軸 | 選択肢 → 軸値 |
|---|---|---|
| Q1 最近、楽しいと感じるのはどんな時間？ | A | 「本や記事をじっくり読む時間」→A1 / 「何かを作ったり手を動かす時間」→A2 / 「人と話したり聞いたりする時間」→A3 |
| Q2 最近、気になっているのはどんなこと？ | B | 「自分の気持ちや内面のこと」→B1 / 「世の中の出来事や歴史」→B2 / 「専門分野や仕組みのこと」→B3 |
| Q3 最近、変えてみたいと感じるのはどれ？ | C | 「物事の見方や考え方」→C1 / 「できることや使える技」→C2 / 「日々の暮らしや習慣」→C3 |
| Q4 今、それにどのくらい時間をかけられそう？ | D | 「今日、少しだけ」→D1 / 「数週間くらい続けて」→D2 / 「年単位でじっくり」→D3 |

設問文は実機確認後の調整を前提とした初期文面である。

#### ボタン動作

選択肢のいずれかをタップした時点でその問いの軸値が確定し、自動的に次の問いを表示する。確認画面・次へボタンは設けない。第4問の選択肢タップで4軸が出揃い、診断結果画面（セクション9.7）に合流する。

#### result_id の決定

ユーザーの4回の選択それぞれから軸A・軸B・軸C・軸Dの値を直接取得し、`{A値} + {B値} + {C値} + {D値}` を連結して result_id を合成する。固定既定値は使用しない。各問が軸の3値すべてに到達可能な設計のため、ユーザーの選択次第で 3×3×3×3 = 81 通りすべての result_id に到達しうる。既存の81種類の result_id 体系・81名のガイド・81通り(=243件)のおすすめルートはそのまま参照し、results.json・guides.json・routes.json の追加は不要である。

#### 行動傾向タグ

topic 診断では behavior_tag を一切付与しない。ガイド判定(セクション12・23)は軸属性(最大4点)のみで行い、12.5の最低スコア保証(2点未満ならデフォルトガイド5名から選出)に依拠する。

#### 結果画面の理由欄

クイズ・今の気分から見つけると同じく2項目方式とする。「選択した目的」「診断形式:気になることから探す」を表示し、3項目目は表示しない。

## 10. 診断ロジック

### 10.1 基本方針

診断結果は user_type ではなく session_mode として扱う。

```text
session_mode = 今回の診断セッションにおける学び方の入口
```

同一ユーザーでも、日によって異なる結果が出てよい。

新仕様では、result_id は4つの軸(A・B・C・D)の組み合わせで定義される81通りの値を取る。

### 10.2 4軸の定義

#### 軸A:関心の対象(行為の場で区別)

軸Aはユーザーが学びの場でどう行為するかを示す。A1は内なる場(自分の頭の中で起こる活動、本を読む・考える・調べる・想像する)、A2は物との場(物理的対象との相互作用、手を動かす・作る・操作する・書く・描く)、A3は人との場(他者との相互作用、話す・聞く・教える・観察する)を意味する。

#### 軸B:知ろうとする対象

軸Bはユーザーが何を知ろうとしているかを示す。B1は自分のこと(自己理解、自分の傾向、自分の欲求)、B2は世界のこと(社会、自然、歴史、文化)、B3は技術のこと(専門知識、特定分野、実用情報)を意味する。

#### 軸C:変えたい方向

軸Cはユーザーが何を変えたいかを示す。C1は考え方を変える(視点、価値観、思考枠組み)、C2はできることを増やす(技能、手順、表現力)、C3は生き方を整える(習慣、関係性、心の状態)を意味する。

#### 軸D:関わり方の深さ

軸Dはユーザーがどのくらいの時間で関わるかを示す。D1は軽く触れる(今日の数十分)、D2は続けて触れる(数週間から数ヶ月)、D3は深く取り組む(年単位の探究)を意味する。

3×3×3×3=81通りの組み合わせが、result_idの基盤となる。

### 10.3 result_idの命名規則

result_idは軸の値を連結した形式とする。たとえばA1B1C1D1、A2B3C2D1、A3B2C3D3など。81通りすべてがresults.jsonに定義される。

### 10.4 表示名の構文

表示名は「[軸Aの動詞]て[軸Bの対象]に触れ、[軸Cと軸Dを反映した結びの語句]」という構文に従う。

軸Aの動詞表現はA1で「考えて」、A2で「手を動かして」、A3で「人と関わって」とする。軸Bの対象表現はB1で「自分」、B2で「世の中」、B3で「専門」とする。

結びの語句は軸Cと軸Dの組み合わせで決まる。C1×D1は「見方を見直す入口」、C1×D2は「見方を変える学び」、C1×D3は「見方を問い直す探究」、C2×D1は「できることを試す入口」、C2×D2は「できることを増やす学び」、C2×D3は「できることを磨く探究」、C3×D1は「暮らしを見直す入口」、C3×D2は「暮らしを整える学び」、C3×D3は「生き方を整える探究」とする。

なお軸C3のD3のみ、対象語が「暮らし」から「生き方」に変わる。これは深さの段階を表現するための例外である。

### 10.5 結果算出(クイズ・今の気分から見つける（診断形式mood）の場合)

各回答にはresult_id用のスコアと行動傾向タグが設定される。すべての回答後にスコアを合算し、最も高いresult_idを今回の結果とする。

軸別の集計を行う。各軸(A、B、C、D)ごとに、回答に含まれるスコアを集計し、最も高い値を持つ軸値を選ぶ。たとえば軸Aで A1=4、A2=2、A3=1 であれば軸AはA1と判定する。4軸の判定結果を連結したものが result_id となる。

### 10.6 同点時の処理

軸の判定で同点が発生した場合、以下の優先順位で決定する。

第一に、今日の目的(purpose_id)に対応する軸値を優先する。purpose_idと軸の対応は以下の通りである。

| purpose_id | 軸A優先 | 軸B優先 |
|---|---|---|
| life | A1 | B1 |
| curiosity | A1 | B2 |
| hobby | A2 | B3 |
| sidejob | A2 | B3 |
| weakness | A1 | B3 |
| browse | A1 | B1 |

軸Cと軸Dは purpose_id では決まらないため、別ルールで決定する。

第二に、診断形式(diagnosis_type)に対応する軸値を優先する。diagnosis_typeがquizの場合は軸C=C2、軸D=D2を優先する。diagnosis_typeがmoodの場合は軸C=C1、軸D=D1を優先する。diagnosis_typeがgameの場合はゲーム結果に従う(セクション10.7)。

第三に、それでも決まらない場合は、results.jsonの並び順で先にある軸値を優先する。

### 10.7 ゲーム診断のresult_id判定と行動傾向判定

ゲーム診断では、軸A・軸Bと、軸C・軸Dの判定経路を分離する。

#### 軸A・軸Bの判定

ゲーム結果からは判定しない。purpose_idから直接マッピングする(セクション10.6の表を参照)。

#### 軸Cの判定

ゲーム結果から取得した指標を用いて軸Cを判定する。スライドパズルの場合、efficiency_ratio < 1.5 かつ backtrack_count < 15 の場合は軸C=C2(できることを増やす)、efficiency_ratio > 2.0 の場合は軸C=C1(考え方を変える)、give_up_count >= 1 または completed == false の場合は軸C=C3(生き方を整える)とする。

迷路の場合、detour_ratio < 1.3 の場合は軸C=C2(できることを増やす)、detour_ratio > 1.8 の場合は軸C=C1(考え方を変える)、give_up_count >= 1 または completed == false の場合は軸C=C3(生き方を整える)とする。

#### 軸Dの判定

ゲーム結果から取得した指標を用いて軸Dを判定する。completed == false または give_up_count >= 2 の場合は軸D=D1(軽く触れる)、completed == true かつ elapsed_seconds < 60 の場合は軸D=D2(続けて触れる)、completed == true かつ elapsed_seconds >= 60 の場合は軸D=D3(深く取り組む)とする。

#### 行動傾向タグの判定

スライドパズルの場合、efficiency_ratio < 1.3 かつ backtrack_count < 10 で planning タグ、efficiency_ratio > 2.0 かつ elapsed_seconds < 60 で quick_action タグ、efficiency_ratio > 1.5 かつ backtrack_count > 20 で exploration タグ、backtrack_count > 30 かつ elapsed_seconds > 180 で cautious タグ、give_up_count == 0 かつ completed == true で persistence タグ、give_up_count >= 2 で switching タグ、first_move_seconds < 3 で intuition タグを付与する。

迷路の場合、detour_ratio < 1.2 で planning タグ、detour_ratio > 1.8 かつ dead_end_count > 5 で exploration タグ、backtrack_cells > path_length × 0.3 で cautious タグ、elapsed_seconds < 30 かつ detour_ratio > 1.5 で quick_action タグ、give_up_count == 0 かつ completed == true で persistence タグ、give_up_count >= 2 で switching タグ、first_move_seconds < 3 で intuition タグを付与する。

#### 主要タグの選出

複数のタグが付与された場合、優先順位(planning、exploration、quick_action、cautious、persistence、switching、intuition、dialogue)で上位2個を主要タグとして採用する。dialogue タグはゲームからは付与されないが、今の気分から見つける（診断形式mood）から付与される可能性がある。

### 10.8 結果の扱い

表示時は必ず以下の表現にする。「今日の結果では、〇〇の入口が合いそうです。」あるいは軸Dに対応して「今日の結果では、〇〇の学びが合いそうです」「今日の結果では、〇〇の探究が合いそうです」と表現する。

以下の表現は禁止する。「あなたは〇〇タイプです」「あなたの本質は〇〇です」のような固定的な分類は使わない。

### 10.9 topic 診断の result_id 判定

topic 診断(セクション9.11)は4問×4軸構成である。第1問が軸A、第2問が軸B、第3問が軸C、第4問が軸Dに対応する。各問の選択肢タップで、その軸の値がユーザー選択から直接決定される。固定既定値は使用しない。

問いごとの選択肢の数は問いによって異なってよく、3選択肢で軸の3値に1対1対応させてもよいし、4選択肢以上で複数の選択肢を同じ軸値に振り分けてもよい。ただし各問において、対応する軸の3つの値すべてが少なくともひとつの選択肢でカバーされていることを設計要件とする。本プロトタイプ実装では各問とも3選択肢の1対1対応とする(具体的な選択肢と軸値の対応表はセクション9.11を参照)。

4問の選択が出揃った時点で result_id は `{A値} + {B値} + {C値} + {D値}` の連結で合成される。ユーザーの選択次第で 3×3×3×3 = 81 通りすべての result_id に到達しうる。これにより既存の81種類の result_id 体系・81名のガイド・81通り(=243件)のおすすめルートを変更せずに、topic 経路から81通りの結果に分岐できる。

topic 診断では behavior_tag を付与しない。ガイド判定は軸属性のみで実施され、12.5の最低スコア保証によりデフォルトガイド群へのフォールバックが必要に応じて働く。

### 10.10 path 診断の result_id 判定

path 診断(セクション9.12)も topic 診断(10.9)と同じく4問×4軸構成である。第1問が軸A、第2問が軸B、第3問が軸C、第4問が軸Dに対応し、各問の選択肢タップで該当軸の値が直接決定される。固定既定値は使用しない。

問いの設問文と選択肢の文面は「これまでの学び」を切り口にしたものとし、topic 診断(「最近の興味・関心」切り口)と切り口が異なる。問いごとの選択肢数は問いによって異なってよく、軸の3値すべてがいずれかの選択肢でカバーされていることを設計要件とする。本プロトタイプ実装では各問とも3選択肢の1対1対応とする(具体的な選択肢と軸値の対応表はセクション9.12を参照)。

result_id の合成方法は topic 診断と同一で、`{A値} + {B値} + {C値} + {D値}` の連結により 81 通りすべての result_id に到達しうる。

path 診断では behavior_tag を付与しない。ガイド判定は軸属性のみで実施され、12.5の最低スコア保証によりデフォルトガイド群へのフォールバックが必要に応じて働く。

## 11. 結果タイプ

### 11.1 result_id一覧の構造

新仕様では81種類のresult_idを定義する。軸A×軸B×軸C×軸Dの3×3×3×3=81通りである。

各 result_id の表示名は、セクション10.4の構文に従って機械的に生成される。たとえば A1B1C1D1 は「考えて自分に触れ、見方を見直す入口」、A2B3C2D2 は「手を動かして専門に触れ、できることを増やす学び」、A3B2C3D3 は「人と関わって世の中に触れ、生き方を整える探究」となる。

完全な81件のリストは data/results.json に定義する。各エントリには result_id、title(表示名)、description(説明文)、recommendations(おすすめ3項目)、avoid(避けたい入口2項目)、guide_id(常に null、ガイドはハイブリッド方式で別途決定)を含む。

### 11.2 result_idの設計原則

第一に、ID命名は軸の値を連結した形式とする(例:A1B1C1D1)。第二に、表示名は四軸すべてを反映する。第三に、description は当該軸の組み合わせに沿った1〜2文の説明とする。第四に、recommendations はパターン辞書方式で生成する3項目で、軸Aと軸Bの組み合わせ、軸Cと軸Dの組み合わせ、軸Bと軸Dの組み合わせから1項目ずつ抽出する。第五に、avoid は軸A逆方向1項目と軸C逆方向1項目の計2項目で構成する。

### 11.3 軸B3×軸C3の組み合わせに関する補足

軸B3(専門)と軸C3(暮らし・生き方)の組み合わせ(9件:A1B3C3D1、A1B3C3D2、A1B3C3D3、A2B3C3D1、A2B3C3D2、A2B3C3D3、A3B3C3D1、A3B3C3D2、A3B3C3D3)では、専門知識と日常生活の橋渡しが必要なため、description文に「日常への応用を意識しながら」という橋渡し句を含める。

### 11.4 デフォルトresult_id

診断結果が算出できない場合のデフォルトは A1B2C1D1(考えて世の中に触れ、見方を見直す入口)とする。

## 12. ガイド対応ルール

### 12.1 基本方針

新仕様では、result_id とガイドは独立した次元で判定する。同じ result_id でも異なるガイドが提示される可能性があり、逆に異なる result_id でも同じガイドが提示される可能性がある。

これは引き継ぎ情報の確定事項「ガイドとresult_idは独立した次元で、それぞれ別の判定ロジックで決定する」に基づく。

### 12.2 ガイドの総数

ガイドは81名を data/guides.json に定義する。各ガイドは古典的人物(おおむね1950年以前死去)に基づく。地域・時代・ジャンルは意図的に分散させる。

### 12.3 ガイドの判定方式(ハイブリッド方式)

ガイド判定は3層構造で行う。

第一層はデータ収集層である。診断形式に応じてユーザーの入力を収集する。ゲーム診断ではゲーム指標、今の気分から見つける（診断形式mood）では選択肢のタグ、クイズ診断では回答時間と自信度を収集する。

第二層は行動傾向タグ生成層である。収集したデータを行動傾向タグ(quick_action、planning、exploration、cautious、dialogue、intuition、persistence、switching の8種類)に変換する。診断形式によって変換ルールは異なる(セクション10.7、セクション9.5、セクション9.4を参照)。

第三層はガイドマッチング層である。ユーザーの行動傾向タグと、result_id の軸組み合わせを入力として、81名のガイドからスコア計算で1名を決定する。

### 12.4 マッチングスコアの計算

各ガイドに対して、以下のスコアを計算する。

第一に、behavior_traits の一致スコア。ガイドの behavior_traits 配列とユーザーの行動傾向タグ集合との一致を計算する。完全一致(全タグが一致)で6点、部分一致(1個のみ一致)で3点、一致なしで0点とする。

第二に、軸属性の一致スコア。ガイドの axis_a、axis_b、axis_c、axis_d 各配列と、result_id の対応する軸値との一致を計算する。各軸が一致するごとに1点(最大4点)を加算する。

第三に、合計スコアを算出し、最高スコアのガイドを選出する。最大は10点となる。同点の場合、過去にユーザーが見たガイドのカテゴリ(ジャンル)とは異なるカテゴリのガイドを優先する。それでも同点の場合はランダムに選出する。

### 12.5 最低スコアの保証

合計スコアが2点以下のガイドしか存在しない場合、汎用性の高いデフォルトガイド群(別途定義する5名)からランダムに選出する。これにより、極端にマッチしないガイドが選ばれることを防ぐ。

デフォルトガイド群は、複数の軸属性を持ち、複数の behavior_traits を持つガイドから選ぶ。具体的には、socrates_dialogue、confucius_learning、leibniz_monad、goethe_synthesis、fukuzawa_independence の5名とする。これらは axis_a、axis_b、axis_c、axis_d、behavior_traits のいずれの値もカバーしうる汎用性を持つ。

### 12.6 ガイド一覧の管理

81名のガイドは9ジャンル×9名で構成する。9ジャンルは哲学・思想、科学・数学、文学・詩、政治・統治、宗教・霊性、芸術・音楽、技術・工学、医療・身体、経済・経営である。

完全な81名のリストは data/guides.json に定義する。各エントリには guide_id、name、name_original、era、region、category、frame_name、display_label、short_line、bio_short、learning_translation、one_min_action、related_topics、axis_a、axis_b、axis_c、axis_d、behavior_traits を含む。

### 12.7 ガイド表現の注意

各ガイドの short_line と learning_translation は、本人の発言や著作物の引用ではなく、考え方を学習選びの文脈に翻訳した独自テキストとする。ガイド詳細画面では「この一言は本人の発言ではなく、考え方を学習向けに要約したものです」という注記を固定表示する。

### 13.5 routes.json

```json
[
  {
    "route_id": "ROUTE_A1B1C1D1_FREE",
    "target_results": ["A1B1C1D1"],
    "route_type": "free",
    "title": "無料で試す:自己理解の入口",
    "description": "まずは短い記事や無料教材で、自己理解の全体像をつかみます。",
    "suitable_for": "今日は軽く知りたい人向け",
    "estimated_time": "5〜10分",
    "url": "https://example.com"
  }
]
```

target_results は新形式の result_id 配列(A1B1C1D1 形式)で指定する。複数の result_id に共通するルートは、target_results に複数の値を含める。

各 result_id に対して、最低3件(無料・低価格・深める)のルートが返るように設計する。81 result_id × 3カテゴリ = 243件以上を確保する。

#### 13.5.1 ルートの設計方針

243件すべてを完全に個別のルートとして作成すると作業量が膨大になる。そのため、以下の階層化方針でルートを管理する。

第一階層は軸B(知ろうとする対象)×軸C(変えたい方向)単位での共通ルート。3×3=9通りのテーマ別ルート(自己×考え方、自己×技能、自己×暮らし、世の中×考え方、…)を、各カテゴリ(無料・低価格・深める)で1〜2件ずつ用意する。これによって基本的なカバレッジを確保する。

第二階層は軸D(深さ)による補完。軸D=D1なら短時間で完結する教材、D2なら数週間継続できる教材、D3なら長期コミット型の教材を、第一階層に追加する形で用意する。

第三階層は軸A(関わり方)による補完。軸A1なら読み物中心、A2なら実作中心、A3なら対話中心の教材を、必要に応じて追加する。

target_results フィールドには、そのルートが対象とする result_id の配列を記載する。同じルートが複数の result_id に対応する場合は、配列に複数の値を含める。

実装時、ある result_id に対応するルートを取得する処理は、routes.json を全件走査し、target_results に当該 result_id を含むルートを抽出する形で行う。

#### 13.5.2 ルート種別

route_type は3種類とする。free(無料で試す)、low_price(低価格で試す)、deep(もう少し深める)である。各 result_id に対して、3種類すべてが最低1件ずつ取得できるよう、routes.json を設計する。

#### 13.5.3 url フィールド

プロトタイプでは仮リンク(https://example.com)でよい。本番版では、ガイドの参考人物に関する書籍へのアフィリエイトリンク、または公開教材へのリンクに置き換える。

### 13.6 データ間の参照関係

新仕様におけるデータ間の参照関係を整理する。

questions.json は独立したデータで、他ファイルへの参照はない。axis_scores は軸の値(A1〜D3)を参照するが、これはコード内の定数として扱う。

quizzes.json も独立したデータで、他ファイルへの参照はない。

results.json は 81件あり、それぞれが result_id を主キーとして持つ。guide_id は常に null である。

guides.json は 81名あり、それぞれが guide_id を主キーとして持つ。result_id への直接参照はない。

routes.json は 243件以上あり、target_results フィールドで results.json の result_id を参照する。

実行時、ユーザーの診断結果からは以下の流れでデータを取得する。第一に診断結果から軸スコアを計算し、result_id を確定する。第二に result_id を用いて results.json から該当エントリを取得し、画面に表示する。第三に行動傾向タグと軸属性を用いて guides.json から最適なガイドをスコア計算で選出する。第四に result_id を用いて routes.json から該当する3〜数件のルートを抽出する。

### 13.7 データ更新時の整合性

新仕様では4つのデータファイル間で参照関係が存在するため、データ更新時には整合性に注意する必要がある。

具体的には、results.json の result_id を変更した場合、routes.json の target_results も同期して更新する必要がある。guides.json の axis_a〜d や behavior_traits を変更した場合、ガイドマッチング結果が変わる。

プロトタイプ段階では手動更新だが、本番版では整合性チェック機能を実装することが望ましい。

## 23. ガイド判定の実装詳細(新規セクション)

### 23.1 判定処理の流れ

ガイド判定処理は以下の流れで実行される。

第一に、診断完了時にユーザーの行動傾向タグ集合を確定する。診断形式によって取得経路が異なる。クイズ診断ではクイズ回答の confidence と response_seconds から、今の気分から見つける（診断形式mood）では回答選択肢の behavior_tag から、ゲーム診断ではゲーム指標からの変換ルール(セクション10.7)から取得する。

第二に、result_id を確定する。クイズ・今の気分から見つける（診断形式mood）では各回答の axis_scores を集計して4軸の値を決定する。ゲーム診断では purpose_id から軸A・軸Bを決定し、ゲーム指標から軸C・軸Dを決定する。

第三に、result_id と行動傾向タグ集合を入力として、ガイドマッチング処理を実行する。

### 23.2 ガイドマッチングの擬似コード

以下は実装の擬似コードである。

```text
function selectGuide(resultId, behaviorTraits):
    bestGuide = null
    bestScore = -1
    
    for each guide in guides.json:
        score = 0
        
        # behavior_traits の一致スコア
        traitMatches = countIntersection(guide.behavior_traits, behaviorTraits)
        if traitMatches >= 2:
            score += 6
        elif traitMatches == 1:
            score += 3
        
        # 軸属性の一致スコア
        axisA = extractAxis(resultId, "A")  # 例: "A1"
        axisB = extractAxis(resultId, "B")
        axisC = extractAxis(resultId, "C")
        axisD = extractAxis(resultId, "D")
        
        if axisA in guide.axis_a:
            score += 1
        if axisB in guide.axis_b:
            score += 1
        if axisC in guide.axis_c:
            score += 1
        if axisD in guide.axis_d:
            score += 1
        
        # 最高スコアを更新
        if score > bestScore:
            bestScore = score
            bestGuide = guide
        elif score == bestScore:
            # 同点処理:ジャンルの分散を優先
            if guide.category != lastSeenCategory:
                bestGuide = guide
    
    # 最低スコア保証
    if bestScore < 2:
        bestGuide = randomSelect(defaultGuides)
    
    return bestGuide
```

defaultGuides は socrates_dialogue、confucius_learning、leibniz_monad、goethe_synthesis、fukuzawa_independence の5名である。

### 23.3 同点処理の詳細

同点が複数発生した場合、第一にユーザーが過去に見たガイドのカテゴリと異なるカテゴリを優先する(localStorage の history を参照)。第二に、それでも同点の場合はランダムに選出する。第三に、historyが空の場合(初回診断時)はランダムに選出する。

### 23.4 行動傾向タグの集計ルール

複数の質問・複数のゲーム結果から複数の行動傾向タグが付与される場合がある。集計ルールは以下の通り。

第一に、同じタグが複数回付与された場合、カウントを増やす。第二に、最終的な集合として上位2タグまでを採用する。第三に、上位2タグの間で同点の場合、優先順位(planning、exploration、quick_action、cautious、persistence、switching、intuition、dialogue)で上位を採用する。第四に、3タグ目以下は破棄する。

### 23.5 デフォルトケースの動作

行動傾向タグが1個も付与されなかった場合(クイズ診断のみで、回答時間や自信度からタグが付与されなかったレアケース)、ガイドマッチングは軸属性のみで行う。この場合、最高スコアは4点となり、最低スコア保証(2点)を満たすケースが大半であるため、デフォルトガイドへのフォールバックは稀である。
