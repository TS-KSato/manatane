# マナタネ (manatane)
## プロトタイプ実装用仕様書 v0.2

---

## 1. 実装目的

サービス名はカタカナで「マナタネ」、英字表記は半角英数小文字で「manatane」とする。
これら以外の表記（MANATANE、Manatane、ManaTane等）は使用しない。

「マナタネ」は、「学びの種」を意味する造語である。

スマートフォン向けの学習入口サービス「マナタネ」の動作プロトタイプを作成する。

本サービスは、ユーザーのその日の目的・気分・興味に合わせて、学び始めるための入口を提案するサービスである。

このプロトタイプの目的は、稟議・企画検討時に以下を確認できる状態にすることである。

- ユーザーが1分程度で診断を体験できる
- 診断結果として「今日の学び方の入口」が表示される
- ガイドキャラクターが表示される
- ガイド詳細ページから別の学習機会へつながる
- おすすめ学習ルートが表示される
- スマートフォン専用UIとして自然に操作できる

本プロトタイプでは、本番機能である会員登録、ログイン、決済、DB保存、外部API連携、アフィリエイト計測は実装しない。

## 6. デザイン方針

### 6.1 基本方針

- スマートフォン専用
- 1画面1アクション
- カード型UI
- タップ中心
- 横スクロールは原則使わない
- 長文を避ける
- 結果はカードで表示する
- 下部固定ナビを配置する
- PCで開いた場合もスマホ幅で中央表示する

### 6.2 画面幅

```css
body {
  max-width: 480px;
  margin: 0 auto;
}
```

### 6.3 ボタン

- 高さは最低48px
- 推奨高さは56px〜64px
- 横幅いっぱいのカード型ボタン
- タップしやすい余白を確保する

### 6.4 色・印象

- 白または淡い背景
- 淡いブルー、グリーン、ベージュ系のアクセント
- 角丸カード
- 過度なアニメーションは禁止
- 占いサイト、高額スクールLP、副業煽りLPのような印象を避ける

### 9.4 クイズ診断画面

#### 目的

短いクイズで興味、自信度、理解状態を取得する。

#### 表示内容

```text
タイトル：
1分クイズ

表示：
・現在の問題番号
・問題文
・選択肢4つ
・回答後に自信度選択
```

#### クイズ数

プロトタイプでは3問出題する。

#### 出題方法

`quizzes.json` から、今日の目的に近いジャンルの問題を3問取得する。
該当ジャンルがない場合はランダムに3問取得する。

#### 回答後の流れ

1. 回答を選ぶ
2. 正誤を表示する
3. 自信度を選ばせる
4. 次の問題へ進む
5. 3問終了後、結果画面へ遷移

#### 自信度選択肢

- かなりある
- たぶんある
- あまりない
- 勘

#### 保存する情報

- `quiz_id`
- `selected_answer_id`
- `is_correct`
- `confidence`

#### スコア算出ルール

クイズ診断におけるスコア算出は以下のルールで行う。

- ユーザーが選んだ選択肢の `scores` を `is_correct` の値に関わらず加算する。
- 理由：`quizzes.json` では不正解の選択肢にも `scores` が設定されており、これは「不正解でも、その選択肢が示すユーザーの傾向（即実行型、深掘り型など）を反映する」ことを意図している。
- `confidence`（自信度）は、プロトタイプ範囲では `result_id` の判定スコアには寄与しない。記録のみ行う。
- 本番版では `confidence` を判定に組み込む可能性があるが、その判定式はプロトタイプ確認後に別途設計する。

## 10. 診断ロジック

### 10.1 基本方針

診断結果は `user_type` ではなく `session_mode` として扱う。

```text
session_mode = 今回の診断セッションにおける学び方の入口
```

同一ユーザーでも、日によって異なる結果が出てよい。

---

### 10.2 結果算出

各回答にはスコアを設定する。

```json
{
  "answer_id": "A_TRY",
  "label": "まず試したい",
  "scores": {
    "first_try": 2,
    "curiosity": 1
  }
}
```

すべての回答後にスコアを合算し、最も高い `result_id` を今回の結果とする。

---

### 10.3 同点時の処理

同点の場合は以下の優先順で決める。

1. 今日の目的（`purpose_id`）に対応する `result_id`
2. 診断形式（`diagnosis_type`）に対応する `result_id`
3. `results.json` の並び順で先にある `result_id`

---

### 10.4 purpose_id と result_id の対応

同点時の判定および結果の補正に使う対応関係を以下に定義する。

| purpose_id  | 対応する result_id     | 対応の根拠                           |
| ----------- | ------------------ | ------------------------------- |
| `life`      | `life_practical`   | 「生活に役立てたい」と「生活に使う入口」が直接対応       |
| `curiosity` | `free_explore`     | 「知的好奇心」と「気ままに探索する入口」が親和         |
| `hobby`     | `deep_understanding` | 「趣味を深めたい」と「じっくり理解する入口」が親和       |
| `sidejob`   | `first_try`        | 「副業の種を探したい」は試行錯誤型で「まず試す入口」と親和   |
| `weakness`  | `step_by_step`     | 「苦手を減らしたい」は段階的取り組みが有効で「手順型」と親和  |
| `browse`    | `free_explore`     | 「今日は眺めるだけ」と「気ままに探索する入口」が直接対応    |

---

### 10.5 diagnosis_type と result_id の対応

同点時の判定の第二優先として使う対応関係を以下に定義する。

| diagnosis_type | 対応する result_id     | 対応の根拠                       |
| -------------- | ------------------ | --------------------------- |
| `quiz`         | `deep_understanding` | 知識確認型の診断は「じっくり理解する入口」と親和   |
| `mood`         | `free_explore`     | 気分・占い型の診断は「気ままに探索する入口」と親和  |
| `game`         | `first_try`        | ゲーム体験型の診断は「まず試す入口」と親和      |

---

### 10.6 結果の扱い

表示時は必ず以下の表現にする。

```text
今日の結果では、〇〇の入口が合いそうです。
```

以下の表現は禁止。

```text
あなたは〇〇タイプです。
あなたの本質は〇〇です。
```

## 19. 完成条件

以下を満たせばプロトタイプ完成とする。

- スマートフォン幅で自然に表示される
- ホームから診断結果まで遷移できる
- 3種類の診断形式を選べる
- 少なくともクイズ診断が正常に完了する
- 少なくともミニゲーム1種類が動作する
- 結果画面に `result_id` に応じた内容が表示される
- 結果に対応したガイドが表示される
- ガイド詳細画面が表示される
- おすすめルートが3件表示される
- localStorageに直近結果が保存される
- 会員登録、決済、DBを使っていない
